# 公司电力市场预测与决策模型研发计划及管理指南

**版本：V1.0**  
**适用范围：电价预测、负荷预测、新能源出力预测、售电决策、储能决策、交易辅助决策等模型研发工作**  
**适用对象：算法研发、数据工程、业务分析、交易运营、项目交付及管理人员**

---

## 一、编制背景

随着国内电力市场化改革加速推进，现货市场、中长期交易、辅助服务、绿电交易、虚拟电厂、独立储能等业务快速发展，电价波动、负荷不确定性、新能源出力不确定性和交易规则复杂性显著提升。公司在售电公司服务、储能运营、电价预测、交易辅助决策等方向的业务需求不断增加，传统依赖人工经验和单一预测模型的方式已经难以支撑规模化、标准化和可复用的项目交付。

目前算法研发工作中仍存在一些典型问题：不同人员各自开发模型，数据口径不统一；模型缺少统一评估标准，难以横向比较；实验结果难以复现；代码复用度不足；模型研发与业务决策之间衔接不够紧密；预测结果尚未充分转化为交易收益、风险控制和运营决策能力。

因此，有必要建立一套公司级模型研发计划和管理标准，推动算法研发从“单点模型开发”转向“数据—模型—决策—复盘”的闭环体系，形成可复用、可验证、可迭代、可交付的模型研发能力。

---

## 二、总体目标

公司模型研发体系的总体目标是：

> 建立面向电力市场业务的标准化模型研发与决策支持体系，使预测模型能够稳定服务于售电报价、批发采购、储能充放电、现货交易、风险预警和经营复盘等实际业务场景。

具体目标包括：

1. **统一研发流程**  
   建立从需求定义、数据治理、特征工程、模型训练、模型评估、回测验证、上线运行到复盘迭代的标准流程。

2. **统一数据与特征口径**  
   对不同省份、不同市场、不同业务主体的数据进行规范化管理，明确字段含义、时间粒度、可用时点、缺失处理和版本管理方式。

3. **统一模型评价标准**  
   不再只用 MAE、RMSE 等单一误差指标评价模型，而是结合曲线形态、价格方向、尖峰/地板价识别、收益回测、风险指标等多维度评价模型价值。

4. **统一代码和实验管理规范**  
   提高代码复用性、实验可复现性和项目交付效率，避免重复开发、口径混乱和结果不可追溯。

5. **推动预测模型接入决策模型**  
   将电价预测、负荷预测、新能源预测结果进一步转化为售电公司报价建议、采购比例建议、储能充放电计划和风险预警结果。

6. **形成公司级模型资产库**  
   沉淀通用数据处理模块、特征工程模块、基准模型、评估工具、回测框架、业务报告模板和项目交付标准。

## 三、研发原则

### 3.1 业务目标优先原则

模型研发必须围绕明确业务问题展开，而不是单纯追求算法复杂度或指标提升。每个模型项目启动前，应明确回答以下问题：

- 服务哪个省份或市场？
- 服务哪个业务主体？
- 决策场景是什么？
- 模型输出会影响什么业务动作？
- 如何衡量模型是否创造价值？

例如，电价预测模型的目标不应仅表述为“降低预测误差”，而应进一步明确其用于售电报价、储能套利、采购比例决策、风险预警还是交易复盘。

### 3.2 可复现原则

所有模型实验必须保证结果可复现。至少应保留：

- 数据版本；
- 特征版本；
- 训练时间范围；
- 验证集和测试集划分；
- 模型参数；
- 随机种子；
- 训练日志；
- 评估结果；
- 预测结果文件；
- 代码提交记录。

任何无法复现的实验结果，不应作为正式结论或项目交付依据。

### 3.3 可比较原则

所有模型必须与统一基准模型进行比较。不能只展示单个模型结果，也不能仅凭个别样本或个别日期判断模型优劣。

每个项目至少应包含：

- 朴素基准；
- 业务经验基准；
- 传统机器学习基准；
- 当前主模型；
- 可选深度学习模型；
- 事后最优或理论上界参考。

### 3.4 严格防止信息泄露原则

电力市场预测和交易决策对时间信息极其敏感。所有模型研发必须严格遵守信息可得性原则：

> 在预测某一时点时，只能使用该时点之前业务上真实可获得的信息。

例如：

- 日前决策不能使用 D 日实际负荷、实际新能源出力或事后发布数据；
- 实时 6 小时提前预测不能使用预测时点之后才公布的数据；
- 样本归一化不能使用测试期或未来数据计算参数；
- 外生变量若存在发布时间滞后，必须按照真实发布时间处理。

发现信息泄露的模型，即使测试指标较好，也不得进入正式评估和业务试用。

### 3.5 预测服务决策原则

公司模型研发不应停留在“预测结果展示”，而应逐步形成“预测—策略—收益—风险—复盘”的闭环。

预测模型输出应尽量包含：

- 点预测；
- 分位数预测；
- 价格区间；
- 高价概率；
- 地板价概率；
- 价格形态分类；
- 日前—实时价差方向；
- 多场景价格曲线；
- 模型置信度。

这些输出比单一预测值更适合接入决策模型。

## 四、模型研发总体流程

公司模型研发建议统一采用以下流程：

```text
业务需求定义
    ↓
数据盘点与口径确认
    ↓
样本构造与特征工程
    ↓
基准模型开发
    ↓
主模型研发与调参
    ↓
统一评估与横向比较
    ↓
决策回测与业务验证
    ↓
模型试运行
    ↓
上线监控与定期复盘
    ↓
版本迭代
```

## 五、业务需求定义标准

每个模型项目启动前，应形成《模型研发需求说明》。内容至少包括：

| 项目 | 说明 |
|---|---|
| 项目名称 | 如“山东实时电价 6 小时提前预测模型” |
| 业务场景 | 售电报价、储能申报、现货风险预警等 |
| 目标市场 | 山东、江苏、重庆、蒙西等 |
| 预测对象 | 日前价格、实时价格、负荷、新能源出力、价差等 |
| 时间粒度 | 15 分钟、1 小时、日度等 |
| 预测周期 | D+1、6 小时提前、月度、年度等 |
| 决策用途 | 报价、采购、充放电、风险控制等 |
| 评价指标 | 预测误差、形态指标、收益指标、风险指标 |
| 交付形式 | 模型代码、预测文件、日报、系统接口、报告等 |

所有需求必须先定义业务用途，再定义技术方案。

## 六、数据管理规范

### 6.1 数据分层管理

建议公司数据统一分为四层：

| 数据层级 | 说明 |
|---|---|
| 原始数据层 | 从交易中心、调度、气象、企业系统、人工文件等来源获取的原始数据 |
| 清洗数据层 | 完成格式统一、时间对齐、异常处理后的数据 |
| 建模数据层 | 已完成特征构造、标签生成、样本切分的数据 |
| 结果数据层 | 模型预测结果、回测结果、交易建议、复盘结果 |

不同层级数据不得混用。建模必须基于稳定版本的建模数据层。

### 6.2 时间字段标准

所有时序数据必须统一包含标准时间字段：

- `datetime`：精确时间戳；
- `date`：日期；
- `time_point`：日内时点；
- `hour`：小时；
- `minute`：分钟；
- `market_day`：交易日；
- `publish_time`：数据发布时间；
- `available_time`：模型可用时间。

其中，`available_time` 对防止信息泄露尤为重要。凡是用于预测或决策的外部数据，均应明确其真实可用时间。

### 6.3 数据版本管理

每个项目必须记录数据版本。建议命名方式如下：

```text
省份_市场类型_数据主题_起止日期_版本号
```

例如：

```text
山东_实时现货_价格负荷边界条件_20250101_20251231_v1.0
江苏_日前实时_价格及出力数据_20250601_20260228_v1.0
```

数据版本一旦用于正式实验，不得直接覆盖修改。若需要修正，应生成新版本并保留变更说明。

### 6.4 缺失与异常处理规范

所有缺失值和异常值处理必须记录规则。常见处理方式包括：

- 删除；
- 前向填充；
- 后向填充；
- 插值；
- 同时刻历史均值填充；
- 相似日填充；
- 模型预测填充；
- 标记缺失 mask；
- 降级使用替代字段。

严禁在未说明的情况下直接填 0、直接删除大段数据或人为修改价格尖峰、地板价等关键市场信号。

## 七、特征工程规范

### 7.1 特征分类

电力市场模型特征建议统一分为以下几类：

| 特征类型 | 示例 |
|---|---|
| 时间特征 | 小时、星期、节假日、月份、季节 |
| 价格特征 | 历史日前价、实时价、价差、峰谷价差 |
| 负荷特征 | 系统负荷、分区负荷、用户负荷 |
| 新能源特征 | 风电、光伏、新能源总出力、预测偏差 |
| 机组与供需特征 | 竞价空间、备用、联络线、检修、开机容量 |
| 气象特征 | 温度、湿度、辐照度、风速、降水 |
| 市场规则特征 | 价格上下限、交易品种、结算规则 |
| 用户侧特征 | 用电曲线、峰谷占比、负荷稳定性 |
| 决策相关特征 | 合约价格、合约比例、SOC、可调节能力 |

### 7.2 特征可用性检查

每个特征进入模型前，必须确认：

- 该特征是否真实可获得；
- 是否存在发布时间滞后；
- 是否包含未来信息；
- 是否在测试期稳定存在；
- 是否存在大面积缺失；
- 是否跨省口径一致；
- 是否对业务决策有解释价值。

不满足要求的特征不能直接进入正式模型。

### 7.3 特征版本管理

建议为每个项目维护特征清单文件，例如：

```text
features_v1.yaml
features_v2.yaml
features_exog_only.yaml
features_no_price_leakage.yaml
```

特征清单中应包含：

- 字段名；
- 字段含义；
- 数据来源；
- 时间粒度；
- 可用时点；
- 缺失处理方式；
- 是否参与训练；
- 是否参与决策解释。

## 八、模型开发规范

### 8.1 基准模型要求

每个项目必须先建立基准模型。常见基准包括：

| 基准类型 | 示例 |
|---|---|
| 历史同日基准 | 使用昨日、上周同日、历史均值 |
| 规则基准 | 峰谷规则、分时段均值 |
| 传统模型 | Linear Regression、Random Forest、LightGBM、XGBoost |
| 业务基准 | 人工经验预测、交易员报价、固定采购比例 |
| 事后参考 | 事后最优策略、最优充放电曲线 |

没有基准模型的项目，不应直接进入复杂模型研发。

### 8.2 模型类型选择

不同业务场景应选择不同模型，不宜盲目追求复杂深度学习结构。

| 场景 | 推荐模型 |
|---|---|
| 小样本、结构化特征充分 | LightGBM、XGBoost、CatBoost |
| 长时序、多变量预测 | PatchTST、Transformer、TCN、LSTM |
| 价格形态分类 | LightGBM、CNN、Transformer Encoder |
| 区间预测 | Quantile Regression、NGBoost、分位数 LightGBM |
| 场景生成 | 残差分布建模、Bootstrap、VAE、扩散模型 |
| 售电报价决策 | 枚举仿真、随机规划、风险约束优化 |
| 储能充放电 | 线性规划、混合整数规划、动态规划、强化学习 |
| 多市场联合优化 | 随机优化、鲁棒优化、场景优化 |

第一阶段建议优先采用“可解释、可回测、可快速迭代”的方法，深度学习模型应作为增强方案，而不是唯一方案。

### 8.3 模型输出标准

预测模型不应只输出单一预测值。正式模型建议统一输出以下字段：

| 字段 | 含义 |
|---|---|
| `datetime` | 预测时点 |
| `target_time` | 被预测时点 |
| `y_pred` | 点预测结果 |
| `y_p10` | 10% 分位数预测 |
| `y_p50` | 中位数预测 |
| `y_p90` | 90% 分位数预测 |
| `scenario_id` | 场景编号 |
| `scenario_value` | 场景价格 |
| `high_price_prob` | 高价概率 |
| `floor_price_prob` | 地板价概率 |
| `confidence` | 模型置信度 |
| `model_version` | 模型版本 |
| `data_version` | 数据版本 |

这样可以为后续决策模型提供更完整输入。

## 九、模型评估标准

### 9.1 点预测误差指标

基础误差指标包括：

| 指标 | 含义 |
|---|---|
| MAE | 平均绝对误差 |
| RMSE | 均方根误差 |
| MAPE | 平均绝对百分比误差 |
| Bias | 平均偏差 |
| Quantile Loss | 分位数预测损失 |

这些指标用于衡量预测值与真实值的接近程度，但不能单独代表业务价值。

### 9.2 曲线形态指标

电价和负荷预测不能只看单点误差，还应关注曲线形态。建议加入：

| 指标 | 含义 |
|---|---|
| 日内曲线相关系数 | 衡量预测曲线与真实曲线形态是否一致 |
| 峰值时刻误差 | 衡量最高价时段是否预测准确 |
| 谷值时刻误差 | 衡量最低价时段是否预测准确 |
| 峰谷价差误差 | 衡量波动幅度是否预测准确 |
| 方向准确率 | 衡量相邻时点涨跌方向是否预测正确 |
| 负相关日期占比 | 衡量预测曲线是否出现严重反向 |

对储能和售电决策而言，曲线形态往往比单点误差更重要。

### 9.3 场景识别指标

对于地板价、尖峰价、高波动日等特殊场景，应单独评估：

| 指标 | 含义 |
|---|---|
| 高价召回率 | 高价时段是否被识别出来 |
| 高价精确率 | 模型判断为高价的时段是否真实高价 |
| 地板价召回率 | 地板价时段是否被识别出来 |
| 极端日识别准确率 | 是否识别出极端价格日 |
| 价差方向准确率 | 日前—实时价差方向是否判断正确 |

对于交易业务，错过极端价格往往比普通时段误差更严重。

### 9.4 决策收益指标

接入决策模型后，应重点评估：

| 指标 | 含义 |
|---|---|
| 预期收益 | 模型策略在预测场景下的收益 |
| 实际收益 | 使用真实价格结算后的收益 |
| 收益提升 | 相比基准策略的收益增量 |
| 正收益天数占比 | 策略盈利稳定性 |
| 最大回撤 | 极端不利阶段的累计损失 |
| VaR | 一定置信水平下的潜在损失 |
| CVaR | 极端亏损场景下的平均损失 |
| Regret | 相比事后最优策略少获得的收益 |
| 约束违约次数 | 是否违反业务约束或调度约束 |

正式评估时，预测指标和决策指标必须同时展示。

## 十、决策模型接入规范

### 10.1 决策模型定位

决策模型不是替代人工交易员，而是为业务人员提供可量化、可解释、可回测的辅助决策依据。

决策模型应输出：

- 推荐方案；
- 收益测算；
- 风险提示；
- 敏感性分析；
- 不同策略对比；
- 不确定性说明；
- 人工可调整参数。

### 10.2 售电公司决策模型

售电公司侧优先建设以下模块：

#### 1. 用户风险画像

对用户进行负荷特征、价格暴露和预测风险评估，形成用户分类：

- 稳定基荷型；
- 尖峰暴露型；
- 波动型；
- 可调节型；
- 高风险型。

#### 2. 零售报价测算

对不同合同结构进行收益与风险测算：

- 固定价套餐；
- 市场联动套餐；
- 分时套餐；
- 保底加分成套餐；
- 尖峰保护套餐。

输出推荐报价区间、风险边际和最低可接受价格。

#### 3. 批发采购比例优化

对中长期与现货采购比例进行测算：

- 保守策略；
- 平衡策略；
- 激进策略；
- 风险约束策略。

输出不同采购比例下的预期毛利、亏损概率和极端亏损。

### 10.3 储能决策模型

储能侧优先建设日前充放电辅助决策模型。输入包括：

- 价格预测曲线；
- 价格场景集；
- 储能容量；
- 最大充放电功率；
- 充放电效率；
- SOC 约束；
- 调度指令；
- 辅助服务机会；
- 政策结算规则。

输出包括：

- 推荐充电时段；
- 推荐放电时段；
- 推荐功率；
- 预计收益；
- 最差场景收益；
- 收益为负概率；
- 是否建议保守申报；
- SOC 保留建议。

第一阶段建议采用“规则识别 + 线性规划 + 场景压力测试”的混合方式。

### 10.4 新能源交易决策模型

新能源场站侧重点关注：

- 中长期合约仓位；
- 月内滚动交易；
- 现货风险敞口；
- 功率预测偏差；
- 限电风险；
- 负电价或地板价风险；
- 日前—实时偏差风险。

模型输出应服务于合约比例调整、现货暴露控制和偏差考核管理。

```text
project_name/

  notebooks/

  reports/
    experiment_report.md
    business_report.md

  tests/
    test_data_pipeline.py
    test_feature_engineering.py
    test_model_output.py

  README.md
  requirements.txt
  pyproject.toml
```

### 11.2 代码提交规范

所有代码变更必须通过版本管理工具保存。提交信息建议采用：

```text
类型: 简要说明
```

例如：

```text
feat: add Shandong RT price feature pipeline
fix: correct information leakage in normalization
exp: add LightGBM baseline for Jiangsu floor price detection
refactor: separate forecast and decision modules
docs: update model evaluation standard
```

常见类型包括：

| 类型 | 含义 |
|---|---|
| feat | 新功能 |
| fix | 修复问题 |
| exp | 实验代码 |
| refactor | 重构 |
| docs | 文档 |
| test | 测试 |
| config | 配置修改 |

### 11.3 分支管理规范

建议至少保留：

| 分支 | 用途 |
|---|---|
| main | 稳定版本 |
| dev | 日常开发版本 |
| experiment/* | 实验分支 |
| feature/* | 功能分支 |
| hotfix/* | 紧急修复分支 |

未经验证的实验代码不得直接合并到 `main` 分支。

### 11.4 代码审查要求

以下内容必须经过审查后才能合并：

- 数据处理逻辑；
- 标签生成逻辑；
- 时间切分逻辑；
- 归一化逻辑；
- 特征可用性判断；
- 模型评估脚本；
- 决策回测脚本；
- 对外交付结果生成脚本。

代码审查重点不是代码风格，而是确认是否存在信息泄露、口径错误、不可复现和业务逻辑偏差。

## 十三、上线与试运行规范

模型进入业务试运行前，必须满足以下条件：

1. 数据口径已经确认；
2. 无明显信息泄露；
3. 已完成测试集评估；
4. 已与基准模型比较；
5. 已完成至少一个完整周期的历史回测；
6. 已输出风险说明；
7. 已确定人工复核机制；
8. 已明确适用范围和不适用范围；
9. 已制定异常处理方案。

模型试运行阶段，不建议直接自动执行交易，应先作为辅助建议，由业务人员复核。

## 十四、模型监控与复盘机制

模型上线或试运行后，应定期监控：

| 监控内容 | 说明 |
|---|---|
| 数据是否正常更新 | 是否缺失、延迟、字段变化 |
| 预测误差是否扩大 | 是否出现模型退化 |
| 曲线形态是否失真 | 是否出现峰谷预测错误 |
| 极端价格是否漏报 | 是否错过高价或地板价 |
| 决策收益是否下降 | 策略是否低于基准 |
| 市场规则是否变化 | 是否需要调整模型 |
| 用户负荷是否变化 | 是否出现用电模式漂移 |

建议建立：

- 日度监控；
- 周度复盘；
- 月度模型评估；
- 季度版本迭代；
- 重大市场规则变化专项复盘。

## 十五、研发路线规划

### 第一阶段：标准化与基准建设

目标是解决“数据不统一、模型不可比、结果不可复现”的问题。

重点工作：

- 建立统一数据目录；
- 建立省份数据字典；
- 建立标准样本生成流程；
- 建立 LightGBM 等基准模型；
- 建立统一评估脚本；
- 建立实验记录规范；
- 建立预测日报模板。

阶段成果：

- 每个重点省份至少一个可复现基准模型；
- 每个模型有完整评估报告；
- 每个实验结果可追溯。

### 第二阶段：预测能力提升

目标是提高模型对价格曲线、极端价格和关键场景的识别能力。

重点工作：

- 引入分时段建模；
- 引入价格形态分类；
- 引入高价和地板价识别模型；
- 引入分位数预测；
- 引入场景生成；
- 对比 LightGBM、Transformer、PatchTST 等模型；
- 建立场景分层评估体系。

阶段成果：

- 不同场景下模型效果清晰可解释；
- 形成点预测、区间预测和场景预测能力；
- 形成可复用预测模型库。

### 第三阶段：接入决策模型

目标是将预测结果转化为业务决策建议。

重点工作：

- 建立售电报价测算模型；
- 建立批发采购比例测算模型；
- 建立储能充放电优化模型；
- 建立日前—实时价差风险模型；
- 建立收益回测框架；
- 建立风险指标体系；
- 输出策略建议报告。

阶段成果：

- 形成“预测—策略—收益—风险”闭环；
- 能够对比不同策略收益；
- 能够输出可解释业务建议。

### 第四阶段：产品化与平台化

目标是将模型能力沉淀为公司级产品和交付能力。

重点工作：

- 建设模型服务接口；
- 建设数据自动更新流程；
- 建设预测与决策看板；
- 建设业务报告自动生成模块；
- 建设模型监控系统；
- 建设项目复制模板；
- 建设多省份模型资产库。

阶段成果：

- 支持多个省份快速复制；
- 支持不同客户定制化交付；
- 形成公司可持续迭代的模型平台。

## 十九、风险与控制措施

| 风险 | 表现 | 控制措施 |
|---|---|---|
| 数据质量风险 | 缺失、错位、口径不一致 | 建立数据校验和版本管理 |
| 信息泄露风险 | 测试效果虚高 | 严格按可用时点建模 |
| 模型过拟合风险 | 历史效果好，实盘效果差 | 使用滚动验证和多阶段测试 |
| 市场规则变化风险 | 模型失效 | 建立规则变更复盘机制 |
| 极端价格风险 | 尖峰或地板价漏判 | 单独建立极端场景模型 |
| 业务误用风险 | 超出模型适用范围 | 明确适用边界和人工复核 |
| 工程不可复现风险 | 结果无法追溯 | 统一代码、数据和实验管理 |
| 决策亏损风险 | 策略收益不稳定 | 引入 VaR、CVaR、压力测试 |

## 二十、结语

公司电力市场模型研发的核心方向，不应停留在单一电价预测模型，而应逐步建设面向真实交易场景的综合决策能力。未来模型体系应围绕电力市场业务形成三层能力：

第一层是**预测能力**，包括电价、负荷、新能源、边界条件和市场状态预测；  
第二层是**风险识别能力**，包括极端价格、价差反转、偏差考核、地板价和高波动场景识别；  
第三层是**决策优化能力**，包括售电报价、批发采购、储能充放电、合约配置和交易策略优化。

只有将这三层能力打通，模型研发才能真正从技术研究转化为公司业务竞争力。公司后续应以标准化、可复现、可比较、可回测、可交付为基本原则，持续沉淀数据资产、模型资产和决策资产，形成可复制到多省份、多主体、多场景的电力市场智能决策体系。
